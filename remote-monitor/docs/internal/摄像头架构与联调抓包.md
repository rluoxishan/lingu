# 摄像头架构说明（联调必读）

> **日期**：2026-07-01  
> **来源**：项目现场确认 + Hasun 说明 + 监控页对接口径

---

## 1. 核心结论（监控页视角）

| 项 | 说明 |
| -- | ---- |
| **摄像头** | **独立设备**，与车辆底盘/清扫控制 **不是同一套控制链** |
| **摄像头控制协议** | **HTTP**，由 **摄像机厂商** 定义（如 `setRtmpUrlInterface`） |
| **前端职责** | 点击「开启摄像头」→ **发一次云平台 HTTP 请求** → 拿响应里的 **各路播流地址** → WebRTC 播放 |
| **后台职责** | 收到前端请求后，**内部**按厂商 HTTP 协议开摄像头、分配推流地址，**一次性**把 playUrl 列表返回前端 |
| **2010005 / 2010002** | 走 **车辆** MQTT（远程驾驶、灯光等），**与摄像头无关** |

```
┌─────────────┐   1010001 / 2010005 / 2010002   ┌─────────────┐
│   车辆      │ ◄──────── MQTT ────────────────►│   云平台     │
│  (底盘等)   │                                  │             │
└─────────────┘                                  │  封装开流 API │
                                                 │             │
┌─────────────┐   厂商 HTTP（后台内部调用）       │  一次响应    │
│  摄像机     │ ◄───────────────────────────────►│  各路 playUrl│
│ (独立设备)  │   setRtmpUrlInterface 等         └──────┬──────┘
└─────────────┘                                         │
                                                        │ 一次 HTTP
                                                        ▼
                                                 ┌─────────────┐
                                                 │  监控页      │
                                                 │  WebRTC 播  │
                                                 └─────────────┘
```

**一句话**：前端只调 **云平台开摄像头 API**；厂商 HTTP、推流分配、多路开流 **全是后台在一次请求里做完**，前端 **不直连** `iot.krzhibo.com`，也 **不碰 MQTT 2010004**。

## 2. 与对外 MQTT 2010004 的关系

| 场景 | 摄像头怎么控 | 谁关心 2010004 |
| ---- | ------------ | -------------- |
| **云平台监控页**（本项目前端） | 云平台 HTTP → 后台调厂商协议 → 返回 playUrl | **前端不关心**；2010004 若存在也是云后台内部实现 |
| **第三方直连 EMQX**（对外协议文档） | MQTT `2010004` 下发摄像机 | 协议交付物 §9；换摄像头方案前语义不变 |

监控页开发与联调 **只验证 HTTP 这一条链**；2010004 留在对外 MQTT 协议里给直连方，**不要写进前端对接逻辑**。

## 3. 联调验证：找在线带摄像头的车抓请求

**目标**：确认「**一次前端请求** → 响应含 **各路 playUrl**」的真实 URL 与 JSON 结构。

### 3.1 操作步骤

1. 登录云平台（如 `sztu.lingubot.cn`），找 **在线 + 有摄像头** 的车辆  
2. 打开监控/视频页，点击 **开启摄像头**（或进入页面自动开流）  
3. DevTools → **Network** → 筛选 `Fetch/XHR`  
4. 记录 **触发开流的那一条（或一组）前端请求**：
   - URL、Method、Request Payload（通常含 `vehicleId` 或相机关联 ID）
   - 响应 JSON：**是否一次返回多路 playUrl**，字段名、结构
   - 关流：离开页面或点关闭时是否有 **关流 HTTP**（或全靠无观众超时）
5. **不需要**在前端侧验证 2010004 / EMQX（那是后台黑盒）

### 3.2 记录模板

| 项 | 内容 |
| -- | ---- |
| 开流 API | URL / Method |
| 请求体 | 例：`{ vehicleId, views?: [...] }` |
| 响应体 | 完整 JSON（脱敏） |
| 单路还是多路 | 一次返回 5 个地址 / 每路单独请求（抓包为准） |
| playUrl 字段 | 例：`playUrl` / `streams[].url` / `cameras.front` |
| 关流 API | URL / 触发时机 |

把结果写入 `docs/internal/联调-抓包记录.md`（可自行新建）。

### 3.3 预期对齐关系

| 抓包看到 | 用途 |
| -------- | ---- |
| 云平台开摄像头 HTTP | VideoPanel 唯一对接入口 |
| 响应含多路播放地址 | WebRTC 播放器绑定 front/left/right… |
| 厂商域名仅出现在后台日志 | 确认前端不直连厂商 |
| devicename ≠ vehicleId | 摄像机独立设备（后台映射，前端可不传） |

## 4. 对监控页开发的含义

1. **视频**：**一次**（或抓包确认的最少次数）调云 API「开摄像头」→ 解析响应里 **每路 playUrl** → WebRTC 播放。  
2. **远程驾驶按钮**：走 **车辆** 控制 API / MQTT 转发，**不要**和开摄像头混在同一接口。  
3. **前端不传** `productid` / `pushurl` / `Streams` 等厂商字段——除非云平台 API 文档明确要求；这些由 **后台** 填。  
4. **关流**：离开页面调云关流 API，或依赖后台「无观众 T 秒」自动关（T 待确认）。  
5. **不要缓存 playUrl 跨会话**——再次进入须重新开流。
