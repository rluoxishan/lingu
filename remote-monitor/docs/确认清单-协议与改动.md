## 0. 架构速览

```
前端监控/控制页
    │
    ├── 订阅 dev/pub/{clientId}  ← 1010001 状态
    ├── 发布 dev/sub/{clientId}  → 2010002 / 2010004 / 2010005 / 2010008
    └── 订阅 dev/reply/{clientId} ← 指令回复

车端 Android 上位机
    ├── MQTT ←→ MCU 串口 (A1 行驶 / A2 外设)
    └── MQTT ←→ 中位机 UDP (任务 / 视频 / 云台 / 作业 / 位姿)
```

| 数据通路 | 控制内容 | 上报内容 |
|---------|---------|---------|
| MCU 串口 A1/A2 | 速度、转向、急停、大灯、转向灯、超声、屏幕 | 速度、转向、车辆状态、电量、超声波、光雨 |
| 中位机 UDP | 视频推流、云台、作业机构 | 位姿(xyz+heading)、作业状态、任务信息 |

**MCU A2 帧仅 6 字节，已全部占用。刹车灯/工作灯/喇叭/作业不在 MCU 串口协议范围内。**

## 一、1010001 — 车辆状态上报

- Topic：`dev/pub/{clientId}`
- 方向：车 → 第三方
- QoS：0
- retain：false
- 频率：1Hz
- 回复：无
- 说明：车端以固定频率持续上报。第三方**只订阅不回复**。收到后直接更新监控页面对应 UI 组件（地图位置、状态栏、仪表盘等）。

### 字段清单

| 字段 | 类型 | 必填 | 数据来源 | 状态 | 详细说明 |
|------|------|------|---------|------|---------|
| `position` | String | 是 | GPS / 中位机 | ✅ | 格式为 `"经度,纬度,高程"`，用逗号分隔三个浮点数。经度范围 -180~180，纬度范围 -90~90，高程单位米。示例：`"114.05129,22.508236,39.19"`。当前代码只输出 `"经度,纬度"` 缺少高程，需修正 |
| `position_xyz` | String | 否 | 中位机 VehiclePose→Position | ⚠️ | 地图坐标系下的位姿，格式为 `"x,y,yaw"`。`x` 和 `y` 单位米，`yaw` 单位弧度。此字段独立于 GPS 坐标，用于室内/无 GPS 场景下展示车辆在地图上的位置和朝向。车端 `VehiclePose` 数据类已有 `Position(x,y,z)` 和 `heading` 字段，数据源具备 |
| `workStatus` | Integer | 是 | MCU A7[6] + 故障判定 | ✅/⚠️ | 车辆工作状态，见 §1.1 枚举。当前代码只输出 0/1/2，需扩展为 0-4。第三方依据此字段决定控制台按钮是否可用（如 workStatus=4 急停时不允许遥控） |
| `battery` | Integer | 是 | MCU A8[6] | ✅ | 电量百分比，整数 0-100。MCU A8 帧第 6 字节直接提供此值。100 表示满电，0 表示耗尽 |
| `brakeStatus` | Integer | 是 | MCU A7[6] 推算 | ⚠️ | 制动状态，见 §1.1 枚举。当前 1010001 报文未包含此字段。由 MCU A7[6] 推算：值为 0x03（紧急制动）时 brakeStatus=1，否则为 0。**MCU 只有紧急制动位，不能区分普通减速刹车和紧急制动**，故此字段实际含义是「是否处于紧急制动状态」 |
| `heading` | Number | 否 | 中位机 VehiclePose.heading | ⚠️ | 航向角，单位度，范围 0-360。0 为正北，顺时针增加。当前 1010001 报文未包含此字段。数据源来自中位机 UDP 上报的 `vehicle_pose.heading`，`VehicleStatusProcessor` 已在解析 |
| `speedMps` | Number | 否 | MCU A7[2-3] | ⚠️ | 当前车速，单位 m/s。MCU A7 帧第 2-3 字节为 int16，值为实际速度×1000。正值表示前进，负值表示后退。当前 1010001 报文未包含此字段，但数据源已具备 |
| `taskId` | String | 否 | 任务引擎 | ✅ | 当前执行中的任务唯一标识，字符串类型。无任务时可为 null 或不传。第三方监控页面的"当前任务"栏位使用此字段展示，配合云 API 查询 `taskName` 等补充信息 |
| `taskProgress` | Integer | 否 | 任务引擎 | ⚠️ | 任务完成进度，整数 0-100。0 表示刚启动，100 表示已完成。计算方式：已完成节点数 ÷ 总节点数 × 100。当前车端 TaskExecutor 未做此计算，需新增 |
| `workingStatus` | String | 否 | 中位机 working_status | ⚠️ | 作业状态字符串，直接透传中位机 UDP 上报的 `working_status` 字段值。已知可能值：`"idle"`（无作业）、`"working"`（正在作业）、`"pausing"`（作业暂停）、`"fault"`（作业故障）。第三方可据此在监控页面展示作业状态标签 |
| `faultSummary` | Object | 否 | FaultCodeProcessor | ⚠️ | 故障摘要对象，包含三个子字段：<br>`hasFault` (Boolean) — 当前是否存在任何活跃故障，有故障时前端点亮告警指示灯<br>`faultCount` (Integer) — 当前活跃故障总数<br>`highestFaultLevel` (Integer) — 最高故障等级 1-4，用于决定告警颜色（1-2黄、3橙、4红）。<br>故障详情（如具体故障码、故障描述）如需获取，由第三方另行调用云 API |
| `sensorStatus` | Object | 否 | 中位机 | ⚠️ | 清扫车传感器数据，见 §1.2。**此字段待确认中位机是否支持，如果不支持则不列入 V1.0**。当前未在 1010001 报文中上报 |

### 1.1 枚举

**workStatus**

| 值 | 枚举名 | MCU A7[6] | 含义 | 第三方 UI 行为建议 |
|----|--------|-----------|------|------------------|
| 0 | IDLE | 0x00 空档 | 车辆空闲，无任务 | 所有控制按钮可用 |
| 1 | WORKING | 0x01 自动驾驶 | 任务执行中 | 监控为主，部分控制受限 |
| 2 | FAULT | — | 故障中 | 显示故障信息，遥控受限（非 4 级可遥控，4 级锁死） |
| 3 | CHARGING | — | 充电中 | 当前 MCU 无此状态，预留。如未来接入充电桩检测则启用 |
| 4 | EMERGENCY_STOP | 0x03 紧急制动 | 急停中 | 遥控禁用，必须解除急停后才能恢复 |

> MCU A7[6] 还有 0x02=手动驾驶状态。当前 MQTT workStatus 不单独定义，手动状态暂映射为 IDLE(0)。
> 注意：值 2 (FAULT) 由 `FaultCodeProcessor` 判定，不来自 MCU。值 3 (CHARGING) 需确认中位机或充电模块是否能提供。

**brakeStatus**

| 值 | 含义 | 判定逻辑 |
|----|------|---------|
| 0 | 正常 | MCU A7[6] ≠ 0x03 |
| 1 | 制动中 | MCU A7[6] = 0x03 |

> 此字段反映的是 MCU 的"紧急制动位"，不是普通行车刹车。车辆正常行驶减速、停靠时的刹车不会触发 brakeStatus=1。

**faultSummary**

| 子字段 | 类型 | 说明 | 数据来源 |
|--------|------|------|---------|
| `hasFault` | Boolean | true=存在至少一个活跃故障 | `activeFaults.isNotEmpty()` |
| `faultCount` | Integer | 当前活跃故障数量 | `activeFaults.size` |
| `highestFaultLevel` | Integer | 1-4，取所有活跃故障中的最高等级 | `activeFaults.maxOf { it.fault_level }` |

> 故障等级含义：1-2 级为警告（黄色告警灯），3 级为严重（橙色），4 级为致命（红色，触发急停+锁车）。

### 1.2 sensorStatus（⚠️ 待确认）

> **确认前提**：这些字段全部来自中位机 UDP，MCU 串口协议不包含任何传感器数据（除电量外）。**请确认中位机是否上报以下字段**。如果没有，整个 `sensorStatus` 对象不列入 V1.0。

| 字段 | 类型 | 必填 | 范围/枚举 | 说明 |
|------|------|------|----------|------|
| `waterTankPct` | Integer | 否 | 0-100 | 水箱水位百分比。100=满，0=空。无水车时不传 |
| `dustbinPct` | Integer | 否 | 0-100 | 垃圾箱容量百分比。100=满，0=空 |
| `envTemp` | Number | 否 | -40 ~ +80 | 环境温度，单位°C。来自车外温度传感器 |
| `brushLeft` | String | 否 | 见下 | 左盘刷状态 |
| `brushRight` | String | 否 | 见下 | 右盘刷状态 |
| `brushMain` | String | 否 | 见下 | 主刷/滚刷状态 |
| `suctionNozzle` | String | 否 | 见下 | 吸嘴状态 |

**刷具/吸嘴状态枚举**：

| 值 | 含义 | 第三方展示建议 |
|----|------|-------------|
| `"normal"` | 正常工作中 | 绿色指示灯 |
| `"fault"` | 故障（如堵转、过流） | 红色告警 |
| `"off"` | 未启用 | 灰色/不显示 |
| `"stuck"` | 卡住/堵转 | 红色告警 + 文字提示 |

## 二、2010005 — 遥控车辆

- Topic：`dev/sub/{clientId}`
- 方向：第三方 → 车
- QoS：0
- 频率：**按下 W/A/S/D 期间以 10–30 Hz 持续下发，松开按键立即发一条 STOP**；发布频率不超过 30 Hz，不低于 10 Hz
- 回复：不需要（低时延优先；如需确认指令到达请使用 2010008）
- 说明：此指令用于 WASD 遥控驾驶。车端不回复。由第三方保证下发频率和乱序控制。

### 两种模式兼容规则

车端收到后按以下优先级判断：

1. 如果 `data.command` 存在且非空 → **模式 A（command 模式）**，忽略 `speed`/`angle`
2. 如果 `data.command` 不存在，但 `data.speed` 或 `data.angle` 存在 → **模式 B（speed/angle 物理量模式）**

### 模式 B 车型兼容

MQTT payload 统一为 `speed`（m/s）+ `angle`（rad），**不随车型变化**。车端按车型配置解释并映射：

| 车型 | angle 车端解释 |
| ---- | -------------- |
| 阿克曼 | 前轮转向角（rad） |
| 四转四驱 / 差速 | 底盘转向/旋转指令（rad）；内部换算为各轮速或运动学参数 |
3. 两者都没有 → 丢弃，不发任何 MCU 指令

### 字段定义

| 字段 | 类型 | 必填 | 适用模式 | 状态 | 详细说明 |
|------|------|------|---------|------|---------|
| `command` | String | 模式A必填 | A | ⚠️ | 遥控命令枚举，见 §2.1。大小写敏感，必须全大写。收到不识别的值则丢弃本条 |
| `speedLevel` | Integer | 否 | A | ⚠️ | 速度档位，整数 1-3。`command` 为 FORWARD 或 BACKWARD 时生效；LEFT/RIGHT/STOP 时忽略。默认值 1。1=低速、2=中速、3=高速 |
| `speed` | Number | 模式B必填 | B | ✅ | 底盘速度，单位 **m/s**。正值前进、负值后退。当前车端直接读取后经限幅 **±1.0 m/s** 下发 MCU A1 |
| `angle` | Number | 模式B必填 | B | ✅ | 转向量，单位 **rad**。正值左转、负值右转。当前车端经限幅 **±0.524 rad**（≈30°）后下发 MCU A1 |
| `seq` | Integer | 否 | A+B | ⚠️ | 指令序号，单调递增的正整数。车端缓存 `lastSeq`，收到的 `seq <= lastSeq` 时丢弃本条，防止网络乱序导致旧指令覆盖新指令。建议从 1 开始，每次下发 +1。前端松开按键发 STOP 时也应递增 seq |

> **V1.0.0 定稿**：云台控制 **不** 通过 2010005 内嵌字段，统一走 **2010002 `target=GIMBAL`**（见 §4.4）。

### 2.1 RemoteDriveCommand 枚举（模式 A）

| 枚举值 | 含义 | speedLevel 对应实际速度 | 前端按键 | 说明 |
|--------|------|------------------------|---------|------|
| FORWARD | 前进 | 1→0.3 / 2→0.6 / 3→1.0 m/s | W | 正速度，直行。speedLevel 控制速度大小 |
| BACKWARD | 后退 | 1→-0.15 / 2→-0.3 / 3→-0.5 m/s | S | 负速度，直行。后退速度低于前进速度（安全设计） |
| LEFT | 左转 | speed=0, angle=+0.3 rad | A | 原地左转。speedLevel 不生效 |
| RIGHT | 右转 | speed=0, angle=-0.3 rad | D | 原地右转。speedLevel 不生效 |
| STOP | 停止 | speed=0, angle=0 | 松开所有键 | 立即停车，不下发急停位（A1[6]=0x00）。与 EMERGENCY_STOP 的区别是 STOP 不会锁车 |
| PAUSE | 暂停 | 不下发到 MCU | — | 预留值。当前车端收到 PAUSE 不做任何 MCU 操作。用于第三方在遥控过程中临时挂起下发但不改变车辆状态 |
| EMERGENCY_STOP | 急停 | speed=0, angle=0, emergency=0x01 | — | 与 2010008 同效果。但 2010005 走 QoS=0 不回复，建议急停场景优先用 2010008（QoS=2 有回复确认） |

> **确认**：以上 FORWARD/BACKWARD 各档位速度值是否合适？LEFT/RIGHT 转向角 0.3 rad（≈17°）是否合适？

### 2.2 档位 P/N/R/D 映射

| 档位 | 对应 command | 前端 UI | 说明 |
|------|-------------|---------|------|
| P 驻车 | `STOP` | P 按钮 | speed=0, angle=0。A1[6]=0x00（不下急停位） |
| R 倒车 | `BACKWARD` + `speedLevel: 1` | R 按钮 | 固定低速后退，确保安全 |
| N 空档 | `STOP` | N 按钮 | 同驻车。**确认：MCU 是否有独立 N 档？如没有则 STOP 即可** |
| D 前进 | `FORWARD` + `speedLevel: 1` | D 按钮 | 固定低速前进。驾驶员可在此基础上叠加 speedLevel 调整 |

### 2.3 下发频率与生命周期

```
第三方按下 W 键：
  每 100-200ms 发一条 → { command: "FORWARD", speedLevel: 2, seq: N }
  seq 每次 +1

第三方松开 W 键：
  立即发一条 → { command: "STOP", seq: N+1 }
  停止下发

第三方按下急停按钮：
  发 → { type: "2010008", data: { action: "EMERGENCY_STOP" } }  （不是 2010005）
```

> 车端 McuController 有 Deadman 保护：500ms 内未收到新 A1 指令则自动下发零速。此机制与协议无关，作为车端安全兜底。

### 2.4 示例

```jsonc
// 模式 A：前进中速 + 序号 1001
{ "type": "2010005", "data": { "command": "FORWARD", "speedLevel": 2, "seq": 1001 } }

// 模式 A：左转 + 序号 1002
{ "type": "2010005", "data": { "command": "LEFT", "seq": 1002 } }

// 模式 A：停止 + 序号 1003
{ "type": "2010005", "data": { "command": "STOP", "seq": 1003 } }

// 模式 B：speed/angle 物理量（m/s + rad）
{ "type": "2010005", "data": { "speed": 0.6, "angle": 0 } }
```

## 三、2010008 — 紧急开关

- Topic：`dev/sub/{clientId}`
- 方向：第三方 → 车
- QoS：2（必须确保送达）
- 频率：单次。按下发一条，解除发一条
- 回复：需要（车端处理后 dev/reply 回复结果）
- 说明：独立于 2010005 的急停指令，走 QoS=2 保证送达，且车端会回复确认。**前端急停按钮应走此指令**。

### 字段定义

| 字段 | 类型 | 必填 | 状态 | 详细说明 |
|------|------|------|------|---------|
| `action` | String | 是 | ⚠️ | 操作类型，取值见 §3.1。大小写敏感，必须全大写。收到不识别的值返回 code=400 |

### 3.1 EmergencyAction 枚举

| 枚举值 | 含义 | MCU A1[6] | 车端处理 | 回复内容 |
|--------|------|-----------|---------|---------|
| EMERGENCY_STOP | 紧急停止 | 0x01=紧急制动 | 下发 A1 急停帧（speed=0, angle=0, emergencyState=0x01），与 4 级故障急停走同一条 MCU 通路。**不等同于锁车**（锁车由故障处理器管理） | `{ code: 200, msg: "EMERGENCY_STOP OK" }` |
| RELEASE | 解除急停 | 0x02=底盘恢复 | 下发 A1 底盘恢复帧（speed=0, angle=0, emergencyState=0x02）。注意：只解除本次紧急制动，**不解除故障导致的锁车** | `{ code: 200, msg: "RELEASE OK" }` |

### 3.2 与 2010005、2010002.brake、2010002.remoteControl 的关系

| 指令 | MCU 效果 | QoS | 回复 | 适用场景 |
|------|---------|-----|------|---------|
| `2010008 EMERGENCY_STOP` | A1[6]=0x01 单次 | 2 | 有 | **前端急停按钮（推荐）** |
| `2010005.command EMERGENCY_STOP` | A1[6]=0x01 单次 | 0 | 无 | 遥控驾驶中快速急停 |
| `2010002.brake: 1` | A1[6]=0x01 单次 | 2 | 有 | 参数/设备控制中触发急停 |
| `2010002.remoteControl: 1` | A1[6]=0x01 每 500ms 持续 | 2 | 有 | 需要持续保持急停的场景 |

> 四种方式最终都下发 A1 帧到 MCU，物理效果相同。区别在于 QoS 保证、是否有回复确认、是否持续下发。

### 3.3 示例

```jsonc
// 急停
{ "type": "2010008", "data": { "action": "EMERGENCY_STOP" } }

// 解除
{ "type": "2010008", "data": { "action": "RELEASE" } }
```

## 四、2010002 — 修改参数 / 设备控制

- Topic：`dev/sub/{clientId}`
- 方向：第三方 → 车
- QoS：2
- 频率：事件触发（点击按钮时发一条）
- 回复：需要（5 秒内 dev/reply）

支持两种写法：**flat 字段**（已实现，可直接用）和 **target 模型**（⚠️ 新定义，待确认）。

### 4.1 flat 字段

所有 flat 字段可在一条 2010002 中混合发送，车端按字段逐个处理。

#### headlights — 大灯控制

| 属性 | 内容 |
|------|------|
| 字段名 | `headlights` |
| 类型 | Integer |
| 值域 | 0=关灯 / 1=近光灯 / 2=远光灯 / 3=远景全开（近光+远光同时亮） |
| MCU 映射 | A2 帧字节[2] |
| 代码状态 | ⚠️ 当前只支持 0/1，需放开 2/3。MCU HeadLamp 枚举已定义 HIGH_BEAM(0x02) 和 ALL_ON(0x03) |
| 说明 | 远光和全开会增加功耗，建议在需要时使用，用完及时切回近光或关灯 |

#### turnSignals — 转向灯

| 属性 | 内容 |
|------|------|
| 字段名 | `turnSignals` |
| 类型 | Integer |
| 值域 | 0=全灭 / 1=左转向灯闪烁 / 2=右转向灯闪烁 / 3=双闪（左右同时闪烁） |
| MCU 映射 | A2 帧字节[6] |
| 代码状态 | ✅ 已完整实现 |
| 说明 | 双闪即危险警示灯。车辆故障时中位机也会自动触发双闪。前端双闪按钮和故障自动双闪互不冲突 |

#### ultrasonic — 超声波

| 属性 | 内容 |
|------|------|
| 字段名 | `ultrasonic` |
| 类型 | Integer |
| 值域 | 0=关闭超声波 / 1=开启超声波 |
| MCU 映射 | A2 帧字节[3]，同时设置字节[1]的车灯控制使能位 |
| 代码状态 | ✅ 已实现 |
| 说明 | 超声波开启后，车辆会使用前后左右的超声波传感器进行障碍物检测 |

#### screen — 屏幕

| 属性 | 内容 |
|------|------|
| 字段名 | `screen` |
| 类型 | Integer |
| 值域 | 1=开启屏幕（通电）/ 0=关闭屏幕（断电） |
| MCU 映射 | A2 帧字节[5]。值 1 → A2[5]=0x00（通电）；值 0 → A2[5]=0x01（断电） |
| 代码状态 | ✅ 已实现 |
| 说明 | 控制车端显示大屏的电源状态。关闭屏幕可节省电量 |

#### brake — 单次紧急制动

| 属性 | 内容 |
|------|------|
| 字段名 | `brake` |
| 类型 | Integer |
| 值域 | 1=触发一次紧急制动。传其他值或 0 无效果 |
| MCU 映射 | A1[6]=0x01 单次 |
| 代码状态 | ✅ 已实现 |
| 说明 | **单次急停**。下发一条 A1 紧急制动帧后即结束。和 2010008 EMERGENCY_STOP 效果相同。如果需要持续保持急停，请使用 `remoteControl` 字段 |

#### remoteControl — 持续急停 ⚠️ 最新代码新增

| 属性 | 内容 |
|------|------|
| 字段名 | `remoteControl` |
| 类型 | Integer |
| 值域 | 1=启动持续急停 / 0=停止持续急停 |
| MCU 映射 | A1[6]=0x01，以 500ms 间隔**持续下发**，直到收到 0 |
| 代码状态 | ✅ 最新代码已实现（`McuController.startRemoteControlZeroSpeed()`） |
| 说明 | 与 `brake` 的区别：`brake:1` 只发一次 A1 急停帧；`remoteControl:1` 启动后每 500ms 持续发送，确保车辆一直处于急停状态。适用场景：需要车辆长时间锁死的操作。**必须成对使用**，收到 `remoteControl:0` 后才停止下发 |

### 4.2 target 模型 ⚠️ 全部新定义

当 `data` 中包含 `target` 字段时，忽略 flat 字段，按 target 模型处理。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `target` | String | 是 | 目标设备类型，取值见 §4.3。大小写敏感，必须全大写 |
| `command` | String | 是 | 操作命令。不同 target 支持的命令不同，见各自章节 |
| `params` | Object | 是 | 操作参数。键值对，具体字段由 target 决定。**即使无参数也必须传空对象 `{}`** |

### 4.3 target 枚举

| target | 含义 | 控制通路 | 吉狮清扫车 | 说明 |
|--------|------|---------|-----------|------|
| GIMBAL | 云台摄像头 | 中位机 UDP | 必选 | 控制摄像头的水平/俯仰/变焦，见 §4.4 |
| WORK | 作业机构 | 中位机 UDP | 必选 | 控制清扫/洒水/吸污/卸料，见 §4.5 |
| AUDIO | 喇叭/语音 | 中位机 UDP | 可选 | 控制喇叭鸣笛，见 §4.6 |
| LIGHT | 独立灯光 | 中位机 UDP 或 MCU 升级 | 待确认 | 控制刹车灯/工作灯等 MCU A2 不支持的灯光，见 §4.7 |

### 4.4 GIMBAL — 云台控制 ⚠️

#### 协议概述

用于控制车端 PTZ 云台摄像头。通过中位机 UDP 下发（不走 MCU 串口）。每条指令车端回复确认。

#### command 枚举

| command | 说明 | 行为 | 必填 params | 可选 params |
|---------|------|------|------------|------------|
| `MOVE` | 绝对移动 | 云台以指定速度移动到目标角度位置 | pan, tilt | zoom, panSpeed, tiltSpeed |
| `MOVE_STEP` | 步进微调 | 在当前角度基础上增加增量值。适合按住方向按钮持续微调的场景 | pan 或 tilt（至少一个） | zoom, panSpeed, tiltSpeed |
| `ZOOM_IN` | 变焦放大 | 步进式放大，每发一条放大一级 | — | — |
| `ZOOM_OUT` | 变焦缩小 | 步进式缩小，每发一条缩小一级 | — | — |
| `HOME` | 回中 | 云台回到原点位置（pan=0, tilt=0, zoom=1.0） | — | — |
| `SET_PRESET` | 设置预置位 | 将当前云台位置保存为指定预置位 | preset | — |
| `GOTO_PRESET` | 转到预置位 | 云台快速转动到已保存的预置位 | preset | — |
| `STOP` | 停止移动 | 立即停止云台所有运动 | — | — |

#### params 定义

| 字段 | 类型 | 单位 | 范围 | 默认值 | 说明 |
|------|------|------|------|--------|------|
| `pan` | Number | 度（deg） | -180 ~ +180 | — | 水平角。0=正前方，正值=向右转，负值=向左转。`MOVE` 时为绝对目标位置，`MOVE_STEP` 时为增量值 |
| `tilt` | Number | 度（deg） | -90 ~ +30 | — | 俯仰角。0=水平，负值=向下，正值=向上。`MOVE` 时为绝对目标位置，`MOVE_STEP` 时为增量值 |
| `panSpeed` | Integer | 度/秒 | 1-100 | 10 | 水平旋转速度。值越大转动越快 |
| `tiltSpeed` | Integer | 度/秒 | 1-50 | 5 | 俯仰旋转速度。俯仰速度一般低于水平速度 |
| `zoom` | Number | 倍数 | 1.0-30.0 | 1.0 | 变焦倍数。1.0=广角端，值越大画面越近。`MOVE` 和 `MOVE_STEP` 时可选附带此字段同时变焦 |
| `preset` | Integer | — | 1-4 | — | 预置位编号。最多支持 4 个预置位。`SET_PRESET` 时保存当前位置到此编号，`GOTO_PRESET` 时转动到此编号保存的位置 |

#### 示例

```jsonc
// 云台转到水平 15°、俯角 5°、变焦 1.5x
{ "type": "2010002", "data": { "target": "GIMBAL", "command": "MOVE",
    "params": { "pan": 15.0, "tilt": -5.0, "zoom": 1.5 } } }

// 在当前角度基础上右转 5°
{ "type": "2010002", "data": { "target": "GIMBAL", "command": "MOVE_STEP",
    "params": { "pan": 5.0 } } }

// 变焦放大一级
{ "type": "2010002", "data": { "target": "GIMBAL", "command": "ZOOM_IN", "params": {} } }

// 回中
{ "type": "2010002", "data": { "target": "GIMBAL", "command": "HOME", "params": {} } }

// 将当前位置保存为预置位 2
{ "type": "2010002", "data": { "target": "GIMBAL", "command": "SET_PRESET",
    "params": { "preset": 2 } } }

// 转动到预置位 2
{ "type": "2010002", "data": { "target": "GIMBAL", "command": "GOTO_PRESET",
    "params": { "preset": 2 } } }

// 停止运动
{ "type": "2010002", "data": { "target": "GIMBAL", "command": "STOP", "params": {} } }
```

### 4.5 WORK — 作业机构 ⚠️

#### 协议概述

用于控制清扫车的作业机构（清扫刷、洒水泵、吸污风机、卸料装置）。通过中位机 UDP 下发。**MCU 串口协议完全不含作业机构控制。**

#### command 枚举

| command | 说明 | 行为 |
|---------|------|------|
| `SET` | 执行/停止指定动作 | 按 `params.action` 和 `params.on` 执行 |
| `STOP` | 全停 | 停止所有正在执行的作业动作。不需要传 action 参数 |

#### params 定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `action` | String | SET 时必填 | 作业动作类型，见下表。大小写敏感，必须全大写 |
| `on` | Boolean | 否 | true=启动动作 / false=停止动作。默认 true |

#### action 枚举

| action | 前端按钮 | 对应的物理设备 | 说明 |
|--------|---------|-------------|------|
| `SWEEP` | 清扫 | 清扫刷（盘刷+主刷） | 启动后刷盘下降并旋转，开始清扫路面 |
| `SPRAY` | 洒水 | 喷淋泵 | 启动后喷水泵工作，向路面洒水降尘。通常和清扫联动使用 |
| `SUCTION` | 吸污 | 吸污风机 | 启动后风机运转，将垃圾吸入垃圾箱 |
| `DUMP` | 卸料 | 卸料机构 | 启动后垃圾箱倾斜/打开，卸出垃圾 |

> **确认**：以上 4 个 action 是否覆盖全部作业功能？是否有遗漏（如"清扫+洒水"一键联动）？名称是否正确？

#### 示例

```jsonc
// 启动清扫
{ "type": "2010002", "data": { "target": "WORK", "command": "SET",
    "params": { "action": "SWEEP", "on": true } } }

// 同时启动清扫和洒水（发两条独立指令）
// 第一条：
{ "type": "2010002", "data": { "target": "WORK", "command": "SET",
    "params": { "action": "SWEEP", "on": true } } }
// 第二条：
{ "type": "2010002", "data": { "target": "WORK", "command": "SET",
    "params": { "action": "SPRAY", "on": true } } }

// 停止洒水
{ "type": "2010002", "data": { "target": "WORK", "command": "SET",
    "params": { "action": "SPRAY", "on": false } } }

// 全部停止
{ "type": "2010002", "data": { "target": "WORK", "command": "STOP", "params": {} } }
```

## 五、2010004 — 视频推流

- Topic：`dev/sub/{clientId}`
- 方向：第三方 → 车
- QoS：2
- 频率：开启后每 5s 重复下发同一条（心跳），停止心跳则自动关流
- 回复：需要

### 字段定义

| 字段 | 类型 | 必填 | 状态 | 详细说明 |
|------|------|------|------|---------|
| `status` | Integer | 是 | ✅ | 0=关闭推流 / 1=开启推流。开启时车端向 `url` 推流；关闭时车端停止推流。车端收到 `status=1` 但 `view` 已开启时，如果 `url` 与当前不同则切换推流地址；相同则视为心跳续期 |
| `view` | Integer | 是 | ✅/⚠️ | 摄像头视角编号，见 §5.1。当前代码只接受 1-4，**需扩展为 1-6**（5=环视/6=云台待确认） |
| `url` | String | status=1 必填 | ✅ | 推流地址，完整的 RTMP/RTSP/WebRTC URL。`status=1` 时必填，`status=0` 时可忽略。推流地址变更时车端应无缝切换到新 URL |

### 5.1 CameraView 枚举

| 值 | 枚举名 | 前端 CameraKey | 状态 | 说明 |
|----|--------|---------------|------|------|
| 1 | FRONT | front | ✅ | 前视摄像头。远程驾驶时的默认主画面 |
| 2 | RIGHT | right | ✅ | 右侧摄像头 |
| 3 | REAR | rear | ✅ | 后视摄像头。倒车时切换为主画面 |
| 4 | LEFT | left | ✅ | 左侧摄像头 |
| **5** | **SURROUND** | surround | ⚠️ | 环视/鸟瞰画面。由多路摄像头拼接合成。**待确认车端是否支持** |
| **6** | **PTZ** | ptz | ⚠️ | 云台摄像头画面。云台模式下作为主画面。**待确认是否作为独立 view 还是仅通过 2010002 GIMBAL 控制** |

> **确认**：view=5 和 view=6 是否列入 V1.0 正式版？

### 5.2 心跳机制

| 参数 | 建议值 | 说明 |
|------|-------|------|
| `heartbeatIntervalSec` | 5 秒 | 第三方每 5 秒重复下发同一条 2010004 指令（相同 view/相同 url）。车端以此判断第三方仍在活跃监控中 |
| `maxMissedHeartbeats` | 3 次 | 车端容忍的连续丢失次数。如果超过 3×5=**15 秒**未收到某 view 的心跳，车端自动关闭该 view 的推流，释放摄像头和网络资源 |

> 心跳由**第三方前端**负责定时重发，车端只管检测超时。心跳间隔和丢失次数的具体值待 Hasun 确认。

### 5.3 多路策略

**方案 A（推荐）**：每路摄像头独立一条 2010004，各自有独立的 `url` 和心跳计时。

```
前视: 2010004 { status: 1, view: 1, url: "rtmp://.../front" }
右视: 2010004 { status: 1, view: 2, url: "rtmp://.../right" }
后视: 2010004 { status: 1, view: 3, url: "rtmp://.../rear" }
左视: 2010004 { status: 1, view: 4, url: "rtmp://.../left" }
环视: 2010004 { status: 1, view: 5, url: "rtmp://.../surround" }  ← 需确认
云台: 2010004 { status: 1, view: 6, url: "rtmp://.../ptz" }       ← 需确认
```

**方案 B**：单 URL 多轨（一条流包含多个 Track）。依赖流媒体服务器支持，不推荐。

> **确认**：方案 A 还是方案 B？由 Hasun 决定。

### 5.4 示例

```jsonc
// 开启前视推流
{ "type": "2010004", "data": { "status": 1, "view": 1,
    "url": "rtmp://stream.example.com/live/vehicle001_front" } }

// 心跳（每 5s 重发一条，内容完全相同）
{ "type": "2010004", "data": { "status": 1, "view": 1,
    "url": "rtmp://stream.example.com/live/vehicle001_front" } }

// 15 秒未收到心跳 → 车端自动关闭推流，等效于收到：
{ "type": "2010004", "data": { "status": 0, "view": 1 } }

// 第三方主动关闭推流
{ "type": "2010004", "data": { "status": 0, "view": 1 } }

// 同时开启 5 路（远程驾驶模式）
// 第三方分别发送 5 条 status=1 指令，各自维护心跳
```

## 六、指令回复信封

所有需要回复的指令（2010002、2010004、2010008），车端处理后通过 `dev/reply/{clientId}` 回复。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | Long | 是 | 原样回传下发指令的 `id`，用于第三方做请求-回复匹配 |
| `code` | Integer | 是 | 业务状态码。200=成功；400=参数错误（如缺少必填字段、枚举值非法）；422=语义错误（如关闭未开启的推流）；500=车端内部处理异常 |
| `msg` | String | 是 | 人类可读的描述文本。成功时为 `"ok"`，失败时描述具体原因（如 `"missing view"`、`"unknown target: XXXX"`） |
| `time` | Long | 是 | 车端生成回复时的毫秒时间戳 |
| `type` | String | 是 | 原样回传下发指令的 `type` |
| `data` | Object | 否 | 附加数据，通常为空对象 `{}`。特定指令可在此携带额外信息 |

```jsonc
// 成功回复
{ "id": 1686004858937364480, "code": 200, "msg": "ok",
  "time": 1740469352100, "type": "2010004", "data": {} }

// 参数错误回复
{ "id": 1686004858937364480, "code": 422, "msg": "missing view",
  "time": 1740469352100, "type": "2010004", "data": {} }

// 未知 target 回复
{ "id": 1686004858937364480, "code": 400, "msg": "unknown target: XXXX",
  "time": 1740469352100, "type": "2010002", "data": {} }
```

> 当前车端代码 `sendResponse()` 把 `code` 和 `msg` 嵌套在 `data` 对象内部，这是 **bug**，会在修复后改为顶层字段（如上所示格式）。
