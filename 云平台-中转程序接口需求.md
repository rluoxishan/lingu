# 云平台 — 中转程序接口需求

> **版本**：V1.0  
> **日期**：2026-06-29  
> **关联文档**：[中转程序-第一期设计定稿.md](./中转程序-第一期设计定稿.md)  

---

## 1. 背景与架构

### 1.1 系统位置

```
车端 ──MQTT──► 云平台 ──HTTP+鉴权──► 中转程序 ──HTTP内网──► 北斗系统
```

**中转程序不直连 EMQX / 车端 MQTT**，所有与车辆相关的读写均通过 **云平台 HTTP API** 完成。

### 1.2 中转程序对云平台的依赖

| 中转能力 | 方向 | 需要云 API |
|----------|------|------------|
| F2 定时推送车辆状态给北斗 | 中转读云 | **查车辆最新状态** |
| F3 北斗反控导航 | 中转写云 | **下发即时坐标导航任务** |
| 全程调用 | 中转 ↔ 云 | **机机鉴权** |

### 1.3 第一期规模

- 车辆数量：**少量，约 1～10 台**  
- 按 **`vehicleId`** 区分车辆  
- 状态推送频率：由北斗注册，典型 **1000ms**（1Hz），即每台车约 1 QPS 读状态

## 3. API 能力需求（P0）

云平台需提供 **3 类 HTTP 接口**（可为现有接口文档化，也可为中转新开发；请明确路径与交付时间）。

---

### 3.1 CLOUD-AUTH — 机机鉴权

#### 需求说明

中转为长期运行的后台服务，需使用 **专用机机账号**，不得复用人机登录 Session。

#### 请提供

| 项 | 要求 |
|----|------|
| 鉴权类型 | OAuth2 Client Credentials / API Key / JWT / 请求签名 等（请指定一种） |
| 获取凭证 | 接口 URL、Method、请求/响应 Body |
| 请求头 | 例如 `Authorization: Bearer {access_token}` |
| Token 有效期 | 过期时间、是否支持 refresh |
| 权限范围 | 可读哪些车辆状态、是否可下发导航 |
| 示例 | **至少 1 组 curl**（成功 + Token 过期失败） |

#### 中转侧行为

- 启动时获取 Token  
- 过期前自动刷新  
- 401/403 记日志并指数退避重试  

---

### 3.2 CLOUD-STATUS — 查询车辆最新状态

#### 需求说明

中转按北斗注册的 `frequency`（毫秒）**轮询**各车最新状态，组装后 POST 给北斗。

第一期约 1～10 台车；请支持 **按单车查询**，若支持 **批量查询** 更佳（减少 RTT）。

#### 建议接口形态（路径由云方定义，以下为逻辑）

| 项目 | 建议 |
|------|------|
| 查询键 | 优先 **`vehicleId`**；若仅支持 `clientId` / `iotId` 请说明 |
| 单车 | `GET /.../vehicles/{vehicleId}/status` 或等价 |
| 批量（可选） | `POST /.../vehicles/status/batch`，body: `{ "vehicleIds": ["V001","V002"] }` |

#### 响应字段需求

云 API 字段名可与 MQTT 不同，但需提供 **与下表映射关系**。数据语义参考《云平台-机器人通信协议-内部标准版》**1010001**。

| 字段用途 | MQTT 参考字段 | 类型 | 必填 | 说明 |
|----------|---------------|------|------|------|
| 地图 X | `position_xyz[0]` | number | 是 | 米，地图坐标系 |
| 地图 Y | `position_xyz[1]` | number | 是 | 米 |
| 航向 | `heading` 或 `position_xyz[2]` | number | 建议 | 请说明单位：**度** 或 **弧度** |
| 车辆状态 | `workStatus` | int | 是 | 0 空闲；1 任务中；2 故障；3 充电；4 急停 |
| 电量 | `battery` | int | 是 | 0–100 |
| 当前任务 ID | `taskId` | string | 否 | 无任务时返回 `""` 或省略 |
| 在线状态 | — | boolean | 建议 | 车辆是否在线 |
| 故障摘要 | `faultSummary.hasFault` | boolean | 建议 | 用于北斗 `isAlert` |
| 故障数量 | `faultSummary.faultCount` | int | 否 | — |
| 最高故障等级 | `faultSummary.highestFaultLevel` | int | 否 | — |
| 故障/告警文案 | — | string | 否 | 用于北斗 `alertMsg` |
| 数据时间戳 | `time`（1010001 外层） | long | 建议 | 毫秒，便于判断数据新鲜度 |

#### 响应示例（逻辑结构，字段名可调整）

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "vehicleId": "V001",
    "clientId": "robot-001",
    "online": true,
    "updatedAt": 1740469352000,
    "positionXyz": {
      "x": 21.64,
      "y": 86.28,
      "yaw": 1.42
    },
    "heading": 81.85,
    "workStatus": 1,
    "battery": 85,
    "taskId": "T001",
    "faultSummary": {
      "hasFault": false,
      "faultCount": 0,
      "highestFaultLevel": 1
    },
    "alertMsg": ""
  },
  "timestamp": 1740469352000
}
```

#### 错误与边界（请明确）

| 场景 | 期望 HTTP / 业务码 |
|------|---------------------|
| vehicleId 不存在 | 404 或 code 非 0，请说明 |
| 车辆离线 | 200 + `online: false`，还是 404？ |
| 无最新 Telemetry | 200 + 空 data，还是 503？ |
| 数据延迟 | 相对车端 1010001（1Hz）通常延迟多少？ |

#### 中转 → 北斗 映射关系（供云方理解用途）

| 北斗字段 | 来源 |
|----------|------|
| x, y | positionXyz.x / .y |
| state | workStatus |
| powerLevel | battery（转 double） |
| currentTask | taskId |
| isAlert | faultSummary.hasFault 或等价 |
| alertMsg | alertMsg 或故障描述 |
| direction | heading（单位待与北斗确认） |

### 3.3 CLOUD-NAV — 下发即时坐标导航

#### 需求说明

北斗系统 POST 物理坐标至中转，中转调用本接口，对 **指定车辆** 下发 **立即执行** 的单点导航任务。

语义等价 MQTT **2010001**：`executionMode: IMMEDIATE`，单节点坐标（无站点名）。

#### 建议接口形态

| 项目 | 建议 |
|------|------|
| Method | POST |
| 路径 | 例如 `POST /.../vehicles/{vehicleId}/navigation/immediate`（由云方定） |
| 指定车辆 | **vehicleId**（必填）；若用 clientId 请说明 |

#### 请求字段需求

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| taskId | string | 建议 | 中转生成，格式 `BD-NAV-{timestamp}`；云可校验唯一 |
| x | double | 是 | 地图 X，米 |
| y | double | 是 | 地图 Y，米 |
| heading / yaw | number | 见说明 | 航向；若云侧用四元数 z/w，请给 **换算公式或示例** |
| z | double | 否 | 若 2010001 需要四元数 z |
| w | double | 否 | 若 2010001 需要四元数 w |
| floor | int | 否 | 北斗会传；云若无此概念可忽略 |

#### 请求示例（逻辑）

```json
{
  "taskId": "BD-NAV-1740469352001",
  "x": 21.64,
  "y": 86.28,
  "heading": 82,
  "floor": 1
}
```

或使用四元数（若云 API 要求）：

```json
{
  "taskId": "BD-NAV-1740469352001",
  "executionMode": "IMMEDIATE",
  "taskNodes": [
    {
      "order": 1,
      "taskPoint": "",
      "duration": "0",
      "x": 21.64,
      "y": 86.28,
      "z": 0.0,
      "w": 1.0
    }
  ]
}
```

#### 响应字段需求

| 字段 | 类型 | 说明 |
|------|------|------|
| code | int | 0 表示云侧已受理 |
| message | string | 失败原因 |
| cloudTaskId | string | 云侧任务 ID（若有） |
| acceptedAt | long | 受理时间毫秒 |

#### 响应示例

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "vehicleId": "V001",
    "cloudTaskId": "T-cloud-12345",
    "taskId": "BD-NAV-1740469352001",
    "acceptedAt": 1740469352000
  },
  "timestamp": 1740469352000
}
```

#### 错误与边界（请明确）

| 场景 | 期望行为 |
|------|----------|
| 车辆不存在 | 404 |
| 车辆离线 | 是否仍排队下发？还是 409/503？ |
| 车辆正在执行其他任务 | 覆盖 / 排队 / 拒绝？ |
| 坐标非法 | 400 + 明确 message |
| 下发超时 | 云侧超时时间建议 ≤ 5s（对齐车端 2010001 回复预期） |

#### 同步语义

第一期中转对北斗为 **同步受理**：云 API 返回成功即认为反控成功，**不等待** 车辆到达目标点。请确认云 API 语义与此一致。

## 4. 车辆标识对照（P0）

请提供文档或查询接口，说明以下 ID 的对应关系及使用场景：

| 标识 | 来源 | 中转使用场景 |
|------|------|--------------|
| vehicleId | 云业务主键 | 配置清单、北斗反控、日志 |
| clientId | MQTT `dev/pub/{clientId}` | 与车云协议对齐 |
| iotId / deviceId | ThingsBoard | RPC 参考文档中的设备 ID |

**请一并提供：**

- 第一期联调用的测试 **vehicleId 列表**（或查询方式）  
- 新增/下线车辆时谁维护、中转如何感知（第一期为配置文件手动更新）

## 5. 联调环境（P0）

| 项 | 需要内容 |
|----|----------|
| 环境 | dev / test 的 **baseUrl**（是否 HTTPS、端口） |
